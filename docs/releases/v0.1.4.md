# v0.1.4

`v0.1.4` 建立 Security Assurance Baseline：把已有安全控制的威胁分析、确定性
对抗回归和有界 fuzz 验证纳入持续维护。它不新增 MCP 工具、Provider 或控制面，也
不构成外部安全审计或形式化证明。

## 威胁模型

新增[威胁模型与证据账本](../threat-model.md)，登记资产、信任边界、攻击者假设、
攻击入口、控制措施、验证证据、剩余风险和非保证范围。首批稳定 threat ID 覆盖：

- identifier/schema confusion、字段侧信道和 row policy 绕过；
- 身份与 transaction token 混淆、trusted proxy spoofing；
- 跨租户缓存污染、reload 竞态和 oversized payload。

该账本只登记 `critical` 与 `high` 威胁。它要求每个威胁关联控制措施、验证方式或
明确的剩余风险；threat ID 不复用。

## 对抗回归与 fuzz

critical/high adversarial corpus 使用普通 table-driven/property 测试和 fuzz seed
表达，覆盖已知恶意输入与安全不变量，并随 `make test` 确定性回放。首批定向 fuzz
target 为：

- `FuzzToolPayloadDecodeNormalizeFieldGate`：MCP payload 解码、规范化与字段门禁；
- `FuzzValidatePredicate`：关系代数 IR validator；
- `FuzzCompileNoInjectionAndQuoting`：参数化 SQL codegen 与标识符引用；
- `FuzzTransactionStateMachine`：事务状态机与失败路径清理。

CI 的 `security-fuzz` 作业对四个 target 分别运行有界、无 Docker 的 fuzz smoke；
本地可使用 `make test-fuzz-smoke` 复现。随机 smoke 仅表示该次运行未发现 crash，
不替代确定性 corpus、race、integration 或 e2e 证据。

## 三库证据边界

PostgreSQL、MySQL 与 OceanBase 均在 CI 中运行 provider integration；共享的
transport、配置、tool、engine、cache 与 transaction 控制不等于三种数据库已有
相同的独立实测证据。尤其是：

- 字段授权、mask 与基础 row policy 路径有三库 integration；
- transaction 身份绑定的 MCP e2e 当前仅覆盖 PostgreSQL；
- dialect 引用、schema 解析、数据库错误文本、连接级 timeout、缓存隔离和 reload
  的 provider 特定行为并未全部逐库独立验证。

适用范围与具体剩余风险见[威胁模型](../threat-model.md)和
[Provider 兼容矩阵](../provider-compatibility.md)。

## 已知限制

- row policy 是应用层控制，不能保护绕过本服务的数据库直连或受信 procedure 内部 SQL；
- trusted proxy 模式依赖代理已认证调用方、移除外部身份 header 并正确配置 CIDR/mTLS；
- stdio 不受 HTTP body/header 上限约束，JSON 解析及上游连接资源仍需部署侧限制；
- 进程内 transaction token 与缓存不提供跨进程或跨实例一致性；进程重启会丢失
  transaction 状态；
- mask 不是加密，也不提供对时序、数据库错误差异、合法聚合小样本推断或关联查询的
  差分隐私保证；
- reload 并发测试不是对所有调度交错的形式化证明，多实例配置发布不在当前一致性模型内。
