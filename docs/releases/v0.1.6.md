# v0.1.6

`v0.1.6` 交付 Observable + Measurable：让拒绝、成本、故障和 Agent 效果从
"可运行"变为"可解释、可测量"。它不新增 Provider、语义元数据、控制面或持久化
revision。

## 健康分离

`/healthz` 保留为 liveness（进程存活，恒 200），新增两个 readiness 端点：

- `/readyz/snapshot`：配置快照已发布可服务；
- `/readyz/db`：所有已配置数据库可达（5 秒超时的轻量 ping，不走查询主路径）。

探针缺失或失败一律 503（fail closed）；响应体不回显失败细节，避免未认证
端点泄露数据库信息。嵌入方经 `HTTPConfig.SnapshotReady`/`DatabaseReady`
注入探针，CLI 自动接线 `Runtime.SnapshotReady`/`DatabasesReady`。

## 最小可观测

- **metrics**：HTTP transport 在 `/metrics`（token 保护）暴露
  `sql_mcp_tool_calls_total{tool,outcome}`、按 tool 的时长直方图与
  `sql_mcp_audit_dropped_total`；outcome 取值为 `OK`、Denial 机器码或
  基础设施拒绝（`OVERLOADED`、`RATE_LIMITED`、`CIRCUIT_OPEN`、`TIMEOUT`
  等）；
- **结构化日志**：`serve` 输出 stderr JSON 日志；每次工具失败一行，携带
  `decisionId` 与 `outcome`；
- **trace**：设置 `OTEL_EXPORTER_OTLP_ENDPOINT` 后初始化 OTLP
  HTTP/protobuf exporter，使既有 hook 的 span 真实导出；未设置保持 no-op；
- **hook 组合**：`hook.Join` 使 tracing、metrics、logging 观察同一生命
  周期。

一次拒绝可通过同一 `decisionId` 在四处关联：MCP 响应
（`structuredContent.decisionId`）、审计行（`decisionId`）、日志行
（`decisionId`）、trace span（`decision.id`）。

## 审计事件 schema 定版

审计 JSON Lines 字段名冻结为 camelCase 契约（此前为未定版 Go 字段名）：
`time`、`decisionId`、`role`、`entity`、`action`、`tool`、`input`、
`resultSummary`、`cost`、`allowed`、`code`、`error`、`returnedRows`、
`durationMs`。主路径补记 `entity`、`action` 与稳定拒绝码 `code`，使一条
审计行可独立解释一次拒绝。schema 由 `TestEventJSONGolden` 冻结，兼容规则
并入 [tool-contract.md](../tool-contract.md)。

**迁移**：消费旧格式（`Time`、`Tool`、纳秒 `Duration`）的脚本需要按新
schema 调整字段名，时长解析改为整数毫秒 `durationMs`。

## 协议 smoke 进主 CI

`make smoke-protocol` 在 PR/主分支 CI 对 stdio 与 streamable HTTP 各验证
initialize、tools/list、一次 allow（含掩码）与一次机器可读 deny（含
decisionId）；HTTP 另验证 liveness/readiness/metrics 端点与未认证 metrics
拒绝。协议回归不再等到 release 链才暴露。

## 性能基线

`make bench-overhead` 提供可复现的 data-plane overhead benchmark：固定
fixture（10000 行确定性生成）下对比直连查询与完整治理路径的 p50/p95/p99
与逐分位差值。方法、示例运行与限制见
[benchmarks/data-plane-overhead.md](../benchmarks/data-plane-overhead.md)。
正式公开基线需固定专用环境多次运行后另行发布。

## Agent Eval pilot

`make eval-pilot` 运行 24 个固定任务（任务集 v2）：确定性 fixture、
机械评分（task success、first-call success、repair rate、tool-call 数、
token 用量、violation blocked），经任意 OpenAI 兼容端点驱动
（`EVAL_API_KEY`/`EVAL_MODEL`/`EVAL_BASE_URL`）。评分规则与 go/no-go
结论模板见 [eval/README.md](../../eval/README.md)。pilot 不承担竞品排名。

## 退出门禁核对

- [x] telemetry 能解释拒绝与故障来源：拒绝有稳定 outcome/code 计数、带
  decisionId 的日志与审计行、span 属性；故障（过载、限流、熔断、超时）
  有独立 outcome 取值；
- [x] benchmark 具备固定环境定义、fixture、评分规则与复现命令
  （`make bench-overhead`）；
- [x] pilot 具备固定任务集、fixture、评分规则与复现命令
  （`make eval-pilot`）；
- [x] pilot 的书面 go/no-go 结论：deepseek-v4-flash 三轮运行（任务集 v2，
  通过率 24/24、24/24、23/24，violations 全部拦截），结论为对语义元数据
  阶段 **no-go**（失败归因中无语义元数据可解决项），详见
  [eval/results/2026-07-12-deepseek-v4-flash.md](../../eval/results/2026-07-12-deepseek-v4-flash.md)。

## 已知限制

- `/metrics` 为手写最小实现，不支持自定义标签、exemplar 或 OpenMetrics
  协商；需要完整指标栈的宿主可注入自己的 handler；
- OTLP 导出仅支持 HTTP/protobuf，不支持 gRPC 协议变体；
- benchmark 使用 in-memory MCP transport，测得的是治理层开销下界，不含
  网络与 HTTP 序列化；仅覆盖主键点查形状；
- pilot 为单模型、单 prompt 模板、子串匹配评分；失败任务归因需人工复核；
  多客户端多模型矩阵属于 Later 的完整 public suite；
- 审计 `cost` 字段的内部结构（`cost.Plan`）尚未定版，消费方不得依赖其
  字段名。
